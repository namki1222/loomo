用一个 Claude Code 会话跑后端、另一个跑前端，你很快会撞上一堵墙：这两个会话看不到彼此。后端一改 API，就得**你亲手**把结果复制粘贴到前端会话里。每一次交接都是手动中转。

**claude-tell-bridge 拆掉了这堵墙。** 你的会话变成会互相发消息的队友——后端改完后自己去告诉前端，前端做完自己那份再把结果回传。你只用自然语言对它们说话，协调交给会话自己完成。

```
没有桥接                              有了桥接
─────────                            ─────────
[后端]  完成，API 改了                [后端] ──"API 改了，更新 UI"──► [前端]
   │                                                                    │
   │  ✋ 你来复制                          [后端] ◄──"完成 ✅"───────────┘
   ▼
[前端]  粘贴到这里…                    你：一句话，剩下的会话自己处理
```

每个会话都是**长期存活**的——一个持有该项目历史与上下文的常驻队友，而不是每次任务之间就把一切忘光的一次性 agent。

## 你会得到什么

三个概念，就是全部模型：

- **一个会话 = 一个项目团队。** 一个会话对应一个项目（`shop`、`blog`……）。
- **一个窗格 = 一位常驻的 AI 队友。** 它的名字既是角色也是地址（`server`、`web`、`infra`）。每个窗格里跑着一个长期存活的 Claude Code，持有那个代码库的历史——而不是生成后就消失的子 agent。
- **它们互相对话。** 用自然语言对任意窗格的 Claude 说（"告诉 web 数据结构改了"），它就会中转给对应的队友，队友做完再回复。你从不亲手输入任何消息命令——会话自己搞定。

```
会话 "shop"  窗格 "server"  ┐
会话 "shop"  窗格 "web"     ├── 同一团队 —— 互相对话
会话 "shop"  窗格 "infra"   ┘

会话 "blog"  窗格 "server"  ─── 不同团队 —— 地址不同，绝不串线
```

没有"连接"按钮。**会话开着，本身就是连接。**

**为什么这样做？**

- **问真正的主人，而不是靠猜。** 常驻会话直接查看自己的代码与环境来作答（"你们 nginx 怎么配的？" → 它打开真实的 conf 文件回答），而不是一个新 agent 的猜测。
- **无需重新上手。** 每个窗格在任务之间持续保留项目历史、记忆与上下文。
- **一屏 = 一座控制塔。** 每位队友的工作都看得见；点任意窗格即可直接指挥。
- **异步、非阻塞。** 请求按 key 追踪，发送方不必等待、继续做下一件事，回复稍后作为新消息到达。
- **玻璃箱。** 每一次交流都显示在屏幕上。没有隐藏的消息总线——你随时可以介入。
- **零基础设施。** 无守护进程、无队列、无 MCP 服务器、无 hook。整个工具就是一个 bash 文件。

## 安装

```bash
npm install -g claude-tell-bridge
tell doctor        # 环境检查
```

管理命令的 Tab 自动补全（可选）：

```bash
echo 'eval "$(tell completion)"' >> ~/.zshrc    # bash 用 ~/.bashrc
```

## 快速上手 —— 从零开始

第一次把 Claude Code 搭成这种结构？大约 5 分钟就能让两个 Claude 互相对话。

> 🐣 **第一次用 tmux？** [新手指南](docs/getting-started.md) 用可复制粘贴的步骤带你走完环境准备、tmux 基本操作、第一次对话。下面是精简版。

**1. 创建 —— 注册工作区**

```bash
tell init
```

> `init` 只做**注册**。它写入配置、在每个目录里植入约定——**此时什么都还没运行。** 启动会话是第 2 步。

向导按顺序询问：

1. **要不要一个中枢（管理）会话？** —— 一个替你指挥多个项目的"秘书"Claude。可选、可跳过；先回车跳过，之后用 `tell hub` 再加。
2. **注册项目** —— 依次：**项目名（= 会话名）** → **这个 Claude 的角色（= 窗格名）** → **目录**。每个项目可有多个角色。例如：项目 `demo` → 角色 `api` → `~/work/demo/api` → 角色 `web` → `~/work/demo/web` → 回车 → 回车。

注册时还会**把协作约定插入每个目录的 CLAUDE.md**——这正是接收方 Claude 知道"该如何回复"的依据。

**2. 运行 —— 启动会话**

```bash
tell up            # 启动所有已注册会话（分割窗格 + 启动 claude），有中枢则接入中枢
tell list          # 通讯录 —— 现在你能对话的对象
```

只启动一个：`tell ws demo`（启动并接入）。因为创建和运行分离，重启后一句 `tell up` 就能还原整个团队。

**3. 第一次对话** —— 直接用自然语言对 `api` 窗格的 Claude 说：

```
给 web 发个 ping 测试，看它是否回复
```

你不输入任何消息命令——约定会让 `api` 的 Claude 自己去做。几秒后，`api` 窗格里就会收到回复。从这一刻起，会话之间就能互相派活了——这正是本工具的核心。

## 快速上手 —— 接管已有会话

**已经在跑 Claude Code 会话了？** 原地接管——无需重启，对话上下文保留。

```bash
tell adopt
```

`adopt` 处理两种情况：

**① 接管活动窗格** —— 分屏里已在运行的 Claude：它扫描每个窗格，让你为每个命名角色，把约定插入该目录的 CLAUDE.md（已有则跳过），还能当场发一条"读取约定 + ping"的消息，让约定无需重启即刻加载并完成验证。

**② 接入你正在进行的对话** —— 来自普通终端标签页的 Claude：给出该对话的会话 ID（其目录会从对话记录中自动识别）——若不知道 ID，就给出目录，从它列出的近期对话中挑选。注册后用 `tell ws <项目>` 启动，该窗格会用 `claude --resume` 起来，**原样继续那段对话**，而不是重新开始。

## 中枢：在一个位置指挥全部（以及手机上）

**中枢**是一个替你指挥其他项目的"秘书"Claude 会话。它不写代码——接收你的请求，委派给合适的会话，追踪回复，再汇总上报。

> 对中枢说一句 *"在服务器加个退款 API，在网页加个按钮"*，它就分派给两边，等各自回复后汇总成 **"✅ 服务器：完成 / ✅ 网页：完成"**。

点对点是核心，但项目一多，一个中枢就很值（在 `tell init` 里创建，或之后用 `tell hub` 添加）：

```
        你（本地键盘 或 手机远程控制）
                    │
                [hub]  ← 只做路由、委派、汇总、上报。不写代码。
              ┌─────┼─────────┐
        [proj-a:*]  [proj-b:*]  [proj-c:*]
```

- **全系统只允许一个中枢。** 所有项目都连到同一个中枢地址。已有中枢时再跑 `init`/`adopt` 不会重复询问，只添加项目。（更换：`tell rm <中枢>` 后 `tell hub`。）
- 早上一句 `tell up` 就把中枢和所有项目都启动，并把你带到中枢。
- 你依然可以点任意窗格直接指挥——中枢是便利，不是关卡。
- **在手机上**：把 Claude Code 远程控制对准中枢会话，你就能在任何地方驱动所有项目。这是"急事在身却没带电脑"的解法。

## 生命周期

一个工作区会经历四个阶段。关键在于**创建与运行是分离的**：

```
创建（注册）           运行（启动）           停止                 删除
tell init/adopt   ───►  tell up / ws  ◄───►  tell down   ───►   tell rm
仅写入配置             在 tmux 里启动         终止（保留配置）      配置 + 约定一并移除
此时什么都没跑         启动 claude                                （你的项目文件不受影响）
```

| 阶段 | 命令 | 作用 |
|---|---|---|
| **创建** | `tell init` · `tell adopt` · `tell hub` | 在 `workspaces.conf` 注册 + 插入 CLAUDE.md 约定。**不运行任何东西。** |
| **运行** | `tell up`（全部）· `tell up <会话>`（单个）· `tell ws <会话>`（单个并接入） | 分割窗格 + 启动 claude（设了会话 ID 则 `--resume`） |
| **停止** | `tell down <会话>` · `tell down --all` | 仅终止会话 —— 配置保留，用 `tell up` 还原 |
| **删除** | `tell rm <会话>` | 终止 + 从配置移除 + 剥离约定块 —— 你的项目文件不受影响 |

## 命令参考

你运行的全是**管理命令**——就那么几个。顶部两行消息命令是 **agent 之间**用的，你从不亲手输入。

| 命令 | 作用 |
|---|---|
| `tell up` | **启动所有已注册会话** → 接入中枢（如有）。已启动的会跳过 |
| `tell up <会话>` | 只启动那个会话（后台 —— 用 `tell ws` 接入） |
| `tell up --tabs` | 启动全部，但为**每个会话开一个终端标签页**，而非单个接入窗口（macOS，中枢排最后 = 获得焦点）。iTerm2 = 标签页 / Terminal.app = 标签页（需"辅助功能"权限——没有则回退为新窗口并提示） |
| `tell down <会话>` \| `--all` | **停止** —— 仅终止会话，保留配置（用 `tell up` 还原）。`--all` 只动已注册的会话 |
| `tell ws` | 列出运行中的会话 + 已注册的工作区 |
| `tell ws <会话>` | 引导工作区（分割窗格、标题、启动 claude；设了会话 ID 则 `--resume`）并接入 |
| `tell init` | 设置向导 —— 注册中枢（可选）+ 项目/角色/目录 + 插入 CLAUDE.md 约定 |
| `tell adopt` | 接入已有 Claude —— 接管活动窗格 + 按会话 ID 恢复此前对话 + 插入约定 |
| `tell hub` | 注册管理（中枢）会话 —— **只允许一个**，已有则拒绝 |
| `tell list` | **通讯录** —— 可对话对象 + 约定/运行状态 |
| `tell rm <会话>` | **删除工作区** —— 终止 + 从配置移除 + 剥离已插入的 CLAUDE.md 约定块（备份 `.bak`）。**你的项目文件/代码不受影响。** 会先展示将移除的内容并确认一次 |
| `tell doctor` | 环境检查（tmux / claude / 配置 / 中枢 / 模板） |
| `tell completion` | Shell 补全脚本 —— 往 `.zshrc` 加一行 `eval "$(tell completion)"` |
| `tell help` | 完整命令列表及说明 |

向导（`init`/`adopt`/`hub`）在**目录输入处也支持 Tab 补全**。

退出码：`0` 已发送 · `1` 无目标窗格 · `2` 参数错误

## 配置文件

`~/.config/claude-tell-bridge/workspaces.conf` —— 每行一个窗格：

```
# 会话|角色|目录|会话ID（可选）   ← # 开头为注释
shop|server|~/work/shop/backend
shop|web|~/work/shop/frontend|f3a1b2c4-...
```

- 同一会话名的多行成为它的窗格，自上而下（第一行创建会话，其余分割）。
- **第 4 字段（可选）= Claude 对话会话 ID** —— 设置后，`ws`/`up` 会用 `claude --resume <ID>` 起该窗格以继续对话（`adopt` 会自动填入）。
- 同目录下的 `hub` 文件指向中枢（`会话|角色`）——它是**保证唯一中枢的依据**，由 `tell hub`/`tell rm` 管理，你几乎不用手动改。
- 支持 `~` 家目录展开。
- 环境变量 `TELL_CONFIG_DIR` 可覆盖配置目录（用于测试 / 多套配置）。

## 故障排查

| 症状 | 原因 · 解决 |
|---|---|
| `[tell] no target pane: X:Y` | 会话名或窗格标题不匹配。查看随错误一起打印的 "(available)" 列表；用 `tmux select-pane -T "角色"` 设置窗格标题 |
| **消息发出去了却没有回复** | 十有八九是**约定未加载**。检查该会话的 CLAUDE.md 是否有约定、会话是否读过。快速修复：让该会话"读取 CLAUDE.md，并用收到的 key 回复" |
| 回复只以聊天文本打印，从未回传 | 同一原因 —— 约定第 1 条（"实际运行回复命令"）没加载 |
| 大量 `not in a mode` 输出 | 目标窗格处于 copy-mode（正在滚动）。`tmux send-keys -t <窗格> -X cancel` 后重发 |
| 发送延迟约 10 秒 | 正常 —— 目标窗格有未提交文本，为避免覆盖而等待 |
| 新窗格显示 shell 而非 claude | 该目录不存在，或 `claude` 不在 PATH。运行 `tell doctor` |

## 常见问题

**这跟子 agent（Task）有什么不同？**
子 agent 是每次都重建上下文的一次性个体。这里的窗格是**常驻队友**——记得昨天为什么那样设计的那个会话来回答你。两者不互斥：每个窗格内部可以再用子 agent。

**为什么用 send-keys 而不是 MCP？**
MCP 消息总线需要服务器与 hook 配置，还把对话藏在协议后面。直接往窗格里打字意味着**人看的屏幕和 AI 接收的通道是同一个**——每次交流都可见，而且安装只是一个脚本。

**如果某个会话一直不回复？**
发送方的 Claude 记着 key，过一阵没回就重新询问或向你上报——这是约定的一部分。投递回执在路线图上。

**两个会话里有相同角色名怎么办？**
没问题——地址是 `会话 + 角色` 的组合。但在**同一个**会话内，角色（窗格标题）必须唯一（命中第一个）。

**能连到远程服务器上的会话吗？**
仅在同一个 tmux 服务器内有效。远程的话，SSH 进那台主机的 tmux，在那边用本桥接。

## 安全

- **仅限可信的本地环境。** 由于基于 `send-keys`，任何能访问同一 tmux 服务器的人都能往任意窗格注入消息。相关 key 用于路由，不是身份认证。
- **绝不要用它发送密码、令牌或密钥。** 它们会以明文留在目标窗格的回滚缓冲和对话记录里。凭据请走带权限的通道（scp 等）传递；约定模板里也写明了这条禁令。
- 接收方 Claude 把消息当作"指令"处理，因此 tmux 服务器访问权 = 指挥每个会话的权力。请知悉这一点。

## 工作原理（底层）

使用本工具完全不需要了解以下内容——这是给好奇者和贡献者看的。

在底层，会话之间用一个叫 `tell` 的命令互发消息，它通过 `tmux send-keys` 直接打进目标窗格的输入框。一个请求带一个 6 位相关 key；回复复用同一个 key，好让发送方把回答和请求对上号。接收方 Claude 遵循自动插入的 CLAUDE.md 约定，其核心规则是：**实际运行**回复命令，而不是把"完成！"当聊天文本打出来（那样别的会话永远看不到）。约定模板在 [`templates/`](templates/)，`init`/`adopt` 会替你插入，你也可以手动粘贴。

机制就这么点：往窗格里打字、用 key 打标签、遵循约定。全部源码就是一个易读的 bash 文件。

## 已知限制与路线图

**限制**
- 输入框检测依赖读取 Claude Code 的提示符渲染（`❯`）—— CLI UI 大改时可能需要检查。
- 无投递保证/回执（发后不管 + 基于约定的重新询问）。
- 头部协议目前是韩语。
- 会话名不能含 `=`。

**路线图**
- [ ] 头部国际化（英文协议 + 模板）
- [ ] Homebrew tap
- [ ] `tell status` —— 列出/追踪待处理的 key
- [ ] 可选的投递回执
