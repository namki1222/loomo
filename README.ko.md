# claude-tell-bridge

**Claude Code 세션들이 서로 대화하게 하라.**

[![license](https://img.shields.io/badge/license-MIT-green?style=flat-square)](LICENSE)

백엔드용 Claude Code 세션 하나, 프론트엔드용 하나를 띄워놓고 일해보면 금방 벽에 부딪힌다 — 이 둘은 서로를 못 본다. 백엔드가 API를 바꾸면 그 결과를 **사람이 복사해서** 프론트 세션에 붙여넣어야 한다. 매번 손으로 중계하는 거다.

**claude-tell-bridge는 그 벽을 허문다.** 세션들이 서로 직접 메시지를 주고받는 동료가 된다 — 백엔드가 변경을 마치면 스스로 프론트에게 알리고, 프론트는 자기 일을 한 뒤 결과를 되돌려준다. 너는 그냥 자연어로 말하면 되고, 조율은 세션들이 알아서 한다.

```
브릿지 없이 — 네가 중계자:

  [백엔드]  "완료, API 바뀜"
      │
      │  ✋ 복사 & 붙여넣기
      ▼
  [프론트]  "...여기 붙여넣기"

브릿지와 함께 — 세션들이 알아서 순환:

       ┌──"API 바뀜, UI 반영해줘"──►┐
  [백엔드]                      [프론트]
       └◄──────"완료 ✅"──────────┘

  너: 한 문장, 나머지는 세션들이 처리
```

각 세션은 **장수명**이다 — 그 프로젝트의 이력과 맥락을 계속 들고 있는 상주 동료지, 작업마다 다 잊어버리는 일회용 에이전트가 아니다.

## 무엇을 얻나

개념은 세 가지, 그게 전부다:

- **세션 = 프로젝트 팀.** 세션 하나가 프로젝트 하나다 (`shop`, `blog`, …).
- **패널 = 상주하는 AI 동료.** 패널 이름이 곧 역할이자 주소다 (`server`, `web`, `infra`). 각 패널에서 장수명 Claude Code가 그 코드베이스의 이력을 들고 돈다 — 스폰됐다 사라지는 서브에이전트가 아니다.
- **세션끼리 대화한다.** 아무 패널의 Claude에게 자연어로 부탁하면("web한테 스키마 바뀐 거 알려줘") 알아서 맞는 동료에게 중계하고, 그 동료가 일을 하고 응답한다. 사람이 메시징 명령을 직접 칠 일은 없다 — 세션들이 알아서 한다.

```
세션 "shop"  패널 "server"  ┐
세션 "shop"  패널 "web"     ├── 같은 팀 — 서로 대화
세션 "shop"  패널 "infra"   ┘

세션 "blog"  패널 "server"  ─── 다른 팀 — 주소가 달라 섞이지 않음
```

"연결" 버튼 같은 건 없다. **세션이 떠 있다는 것 자체가 연결이다.**

**왜 이 방식인가?**

- **추측하는 놈이 아니라 주인에게 물어본다.** 상주 세션이 자기 코드·환경을 직접 확인하고 답한다 ("너네 nginx 어떻게 세팅했어?" → 실제 conf를 열어보고 답한다) — 신선한 에이전트의 추측이 아니라.
- **재온보딩 없음.** 각 패널은 프로젝트 이력·메모리·맥락을 작업 간에 계속 유지한다.
- **한 화면 = 관제탑.** 모든 동료가 일하는 게 눈에 보이고, 아무 패널이나 클릭해 직접 지시할 수 있다.
- **비동기·논블로킹.** 요청을 키로 추적하니 보낸 쪽은 기다리지 않고 다음 일을 한다. 응답은 나중에 새 메시지로 들어온다.
- **유리 상자.** 모든 대화가 화면에 그대로 보인다. 숨은 버스 없음 — 언제든 끼어들 수 있다.
- **인프라 제로.** 데몬·큐·MCP 서버·훅 설정이 없다. 전체가 bash 파일 하나다.

## 설치

```bash
npm install -g claude-tell-bridge
tell doctor        # 환경 점검
```

관리 명령용 탭 자동완성 (선택):

```bash
echo 'eval "$(tell completion)"' >> ~/.zshrc    # bash는 ~/.bashrc
```

## 빠른 시작 — 처음부터

Claude Code를 이 구조로 처음 세팅한다면 — 약 5분 안에 두 Claude가 서로 대화하는 것까지 간다.

> 🐣 **tmux가 처음이라면** — 준비물 설치부터 tmux 기본 조작, 첫 대화 확인까지 복붙으로 따라가는 [초보자용 가이드](docs/getting-started.md)를 보세요. 아래는 그 압축판입니다.

**1. 생성 — 워크스페이스 등록**

```bash
tell init
```

> `init`은 **등록만** 한다 — 설정을 기록하고 각 디렉터리에 규약을 심을 뿐, **아직 아무것도 실행되지 않는다.** 세션을 켜는 건 2단계.

마법사가 순서대로 묻는다:

1. **허브(관리자) 세션?** — 여러 프로젝트를 대신 지휘하는 '비서' Claude. 선택이고 건너뛸 수 있다. 처음엔 엔터로 건너뛰고 나중에 `tell hub`로 추가하면 된다.
2. **프로젝트 등록** — 순서대로: **프로젝트 이름(=세션 이름)** → **이 Claude의 역할(=패널 이름)** → **디렉터리**. 역할은 여러 개 가능. 예: 프로젝트 `demo` → 역할 `api` → `~/work/demo/api` → 역할 `web` → `~/work/demo/web` → 엔터 → 엔터.

등록하면 **각 디렉터리의 CLAUDE.md에 협업 규약이 자동 삽입**된다 — 받는 쪽 Claude가 "어떻게 응답해야 하는지" 아는 근거다.

**2. 실행 — 세션 켜기**

```bash
tell up            # 등록된 세션 전부 켜기 (패널 분할 + claude 실행), 허브 있으면 허브로 접속
tell list          # 주소록 — 지금 말 걸 수 있는 상대
```

하나만 켜려면 `tell ws demo` (켜고 바로 접속). 생성과 실행이 분리돼 있어서, 재부팅 후에도 `tell up` 한 번이면 팀 전체가 복원된다.

**3. 첫 대화** — `api` 패널의 Claude에게 그냥 말로 부탁한다:

```
web한테 핑퐁 테스트 보내고 응답 오는지 확인해줘
```

메시징 명령을 직접 칠 필요 없다 — 규약 덕에 `api`의 Claude가 알아서 한다. 몇 초 뒤 `api` 패널에 응답이 돌아온다. 이 순간부터 세션들은 서로에게 일을 시킬 수 있다 — 이게 이 도구의 코어다.

## 빠른 시작 — 이미 쓰던 세션 편입

**이미 쓰던 Claude Code 세션이 있다면** — 재시작 없이, 대화 맥락을 유지한 채 그대로 편입한다.

```bash
tell adopt
```

`adopt`는 두 경우를 처리한다:

**① 떠 있는 패널 편입** — 분할 화면에서 이미 돌던 Claude: 떠 있는 패널을 전부 스캔해서 각각 역할 이름을 정하게 하고, 그 디렉터리 CLAUDE.md에 규약을 삽입하고(이미 있으면 건너뜀), 원하면 그 자리에서 "규약 읽어 + 핑퐁" 메시지를 보내 재시작 없이 로드+검증까지 끝낸다.

**② 쓰던 대화 가져오기** — 일반 터미널 탭에서 쓰던 Claude: 대화의 세션ID만 주면 된다(담당 디렉터리는 대화 기록에서 자동 감지). ID를 모르면 디렉터리를 주고 최근 대화 목록에서 고르면 된다. 등록 후 `tell ws <프로젝트>`로 띄우면 그 패널은 `claude --resume`으로 **기존 대화를 그대로 이어받아** 뜬다.

## 허브: 한 자리에서 전부 지휘 (그리고 폰에서)

**허브**는 다른 프로젝트들을 대신 지휘하는 '비서' Claude 세션이다. 코드를 직접 짜지 않고 — 요청을 받아 맞는 세션에 위임하고, 응답을 추적해 모아서 보고한다.

> 허브에게 *"서버에 환불 API 추가하고 웹에 버튼 달아줘"* 한 마디 → 양쪽에 나눠 시키고, 각 응답이 오면 **"✅ 서버: 완료 / ✅ 웹: 완료"**로 취합해 알려준다.

P2P 대화가 코어지만, 프로젝트가 여럿이면 허브 하나를 두는 게 좋다 (`tell init`에서 만들거나 나중에 `tell hub`로 추가):

```
        너 (로컬 키보드 or 폰 Remote Control)
                    │
                [hub]  ← 라우팅·위임·취합·보고만. 코드는 안 짬
              ┌─────┼─────────┐
        [proj-a:*]  [proj-b:*]  [proj-c:*]
```

- **허브는 시스템 전체에 하나만.** 모든 프로젝트가 같은 허브 주소로 연결된다. `init`/`adopt`를 다시 돌려도 허브가 있으면 안 묻고 프로젝트만 추가한다. (교체: `tell rm <허브>` 후 `tell hub`)
- 아침에 `tell up` 한 번이면 허브·프로젝트가 전부 켜지고 허브로 접속된다.
- 여전히 아무 패널이나 클릭해 직접 지시할 수 있다 — 허브는 편의지 관문이 아니다.
- **폰에서**: Claude Code Remote Control을 허브 세션에 물리면 밖에서도 전 프로젝트를 굴린다. "급한데 노트북이 없다"의 해법.

## 라이프사이클

워크스페이스는 네 단계를 거친다. 핵심은 **생성과 실행이 분리**돼 있다는 것:

```
생성(등록)             실행(켜기)             끄기                 삭제
tell init/adopt   ───►  tell up / ws  ◄───►  tell down   ───►   tell rm
설정만 기록            tmux에 띄움            종료(설정 유지)       설정+규약 제거
아직 아무것도 안 뜸    claude 실행                                (프로젝트 파일 무손상)
```

| 단계 | 명령 | 하는 일 |
|---|---|---|
| **생성** | `tell init` · `tell adopt` · `tell hub` | `workspaces.conf` 등록 + CLAUDE.md 규약 삽입. **실행 안 됨** |
| **실행** | `tell up` (전체) · `tell up <세션>` (하나) · `tell ws <세션>` (하나 켜고 접속) | 패널 분할 + claude 실행 (세션ID 있으면 `--resume`) |
| **끄기** | `tell down <세션>` · `tell down --all` | 세션 종료만 — 설정 유지, `tell up`으로 복원 |
| **삭제** | `tell rm <세션>` | 종료 + 설정 제거 + 규약 블록 제거 — 프로젝트 파일은 무손상 |

## 명령어 레퍼런스

네가 쓰는 건 아래 **관리 명령**이 전부다 — 몇 개 안 된다. 세션끼리 주고받는 메시징 명령은 **에이전트들**이 규약대로 알아서 실행하니 사람이 칠 일이 없다.

| 명령 | 설명 |
|---|---|
| `tell up` | **등록된 세션 전부 켜기** → 허브(있으면)로 접속. 이미 뜬 건 건너뜀 |
| `tell up <세션>` | 그 세션만 켜기 (백그라운드 — 접속은 `tell ws`) |
| `tell up --tabs` | 전부 켜되 한 창 접속 대신 **세션마다 터미널 탭**을 연다 (macOS, 허브가 마지막=포커스). iTerm2=탭 / Terminal.app=탭('손쉬운 사용' 권한 필요 — 없으면 새 창 폴백 + 안내) |
| `tell down <세션>` \| `--all` | **끄기** — 세션 종료만, 설정 유지 (`tell up`으로 복원). `--all`은 등록된 세션만 건드림 |
| `tell layout [<세션>] <프리셋>` | 패널 배치 변경 — `tiled` / `main-vertical` / `main-horizontal` / `even-horizontal` / `even-vertical`. `tmux.conf` 편집 불필요 |
| `tell ws` | 실행 중 세션 + 등록된 워크스페이스 목록 |
| `tell ws <세션>` | 워크스페이스 부트스트랩(패널 분할·제목·claude 실행, 세션ID 있으면 `--resume`) 후 접속 |
| `tell init` | 셋업 마법사 — 허브(선택) + 프로젝트/역할/디렉터리 등록 + CLAUDE.md 규약 삽입 |
| `tell adopt` | 쓰던 Claude 가져오기 — 떠 있는 패널 편입 + 기존 대화(세션ID) 이어받기 + 규약 삽입 |
| `tell hub` | 관리자(허브) 세션 등록 — **하나만 허용**, 이미 있으면 거절 |
| `tell list` | **주소록** — 말 걸 수 있는 상대 + 규약/실행 여부 |
| `tell rm <세션>` | **워크스페이스 삭제** — 종료 + 설정 제거 + 삽입했던 CLAUDE.md 규약 블록 제거(백업 `.bak`). **프로젝트 파일·코드는 무손상.** 삭제 목록 보여주고 한 번만 확인 |
| `tell doctor` | 환경 점검 (tmux / claude / 설정 / 허브 / 템플릿) |
| `tell completion` | 셸 자동완성 스크립트 — `.zshrc`에 `eval "$(tell completion)"` 한 줄 |
| `tell help` | 전체 명령·설명 |

마법사(`init`/`adopt`/`hub`)의 **디렉터리 입력에서도 탭 완성**이 된다.

종료코드: `0` 전송 · `1` 대상 패널 없음 · `2` 인자 오류

## 설정 파일

`~/.config/claude-tell-bridge/workspaces.conf` — 한 줄에 패널 하나:

```
# 세션|역할|디렉터리|세션ID(선택)   ← #으로 시작하면 주석
shop|server|~/work/shop/backend
shop|web|~/work/shop/frontend|f3a1b2c4-...
```

- 같은 세션 이름의 줄들이 위에서부터 패널이 된다 (첫 줄이 세션 생성, 이후 분할).
- **4번째 필드(선택) = Claude 대화 세션ID** — 있으면 `ws`/`up`이 `claude --resume <ID>`로 그 대화를 이어받아 띄운다 (`adopt`가 채워줌).
- 같은 폴더의 `hub` 파일 = 허브 포인터(`세션|역할`). **허브가 하나뿐임을 보장하는 근거**로, `tell hub`/`tell rm`이 관리하니 직접 고칠 일은 거의 없다.
- `~` 홈 확장 지원.
- 환경변수 `TELL_CONFIG_DIR`로 설정 디렉터리를 바꿀 수 있다 (테스트/멀티 프로필용).

## 트러블슈팅

| 증상 | 원인 · 해법 |
|---|---|
| `[tell] 대상 패널 없음: X:Y` | 세션 이름 또는 패널 제목 불일치. 에러에 함께 뜨는 "(가용)" 목록 확인. 패널 제목 지정: `tmux select-pane -T "역할"` |
| **메시지는 갔는데 응답이 안 옴** | 십중팔구 **규약 미로드**. 그 세션 CLAUDE.md에 규약이 있고 세션이 읽었는지 확인. 빠른 해법: 그 세션에 "CLAUDE.md 읽고 받은 키로 응답해줘"라고 보내기 |
| 응답이 채팅 텍스트로만 찍히고 안 돌아옴 | 같은 원인 — 규약 1번("응답 명령을 실제 실행")이 로드 안 됨 |
| `not in a mode` 가 잔뜩 출력 | 대상 패널이 copy-mode(스크롤 중)였음. `tmux send-keys -t <패널> -X cancel` 후 재전송 |
| 전송이 10초쯤 지연됨 | 정상 — 상대 입력창에 미제출 텍스트가 있어 덮어쓰기 방지 대기 중 |
| 새 패널에 claude 대신 셸만 뜸 | 그 디렉터리가 없거나 `claude`가 PATH에 없음. `tell doctor` |

## FAQ

**서브에이전트(Task)랑 뭐가 다른가?**
서브에이전트는 매번 컨텍스트를 새로 쌓는 일회용이다. 여기 패널은 **상주 동료**다 — 어제 왜 그렇게 설계했는지 기억하는 세션이 답한다. 배타적이지 않다: 각 패널이 내부적으로 서브에이전트를 쓰면 된다.

**MCP 대신 왜 send-keys인가?**
MCP 메시지 버스는 서버·훅 설정이 필요하고 대화를 프로토콜 뒤로 숨긴다. 패널에 직접 타이핑하면 **사람이 보는 화면과 AI가 받는 채널이 같아서** 모든 대화가 눈에 보이고, 설치가 스크립트 하나다.

**응답을 안 하면?**
보낸 쪽 Claude가 키를 기억하고 있다가 한참 없으면 재요청하거나 사용자에게 보고한다 — 규약의 일부다. 전달 영수증은 로드맵에 있다.

**같은 역할 이름이 두 세션에 있으면?**
문제없다 — 주소는 `세션+역할` 쌍이다. 단 **한 세션 안에서** 역할(패널 제목)은 유일해야 한다 (첫 매칭 우선).

**원격 서버의 세션과도 되나?**
같은 tmux 서버 안에서만 동작한다. 원격은 SSH로 그 호스트의 tmux에 들어가 그쪽 브릿지를 쓰는 방식.

## 보안

- **신뢰된 로컬 환경 전용.** `send-keys` 기반이라 같은 tmux 서버에 접근 가능한 누구나 어떤 패널에든 메시지를 주입할 수 있다. 상관키는 라우팅용이지 인증이 아니다.
- **비밀번호·토큰·시크릿을 절대 이걸로 보내지 마라.** 상대 패널 스크롤백과 트랜스크립트에 평문으로 남는다. 자격증명은 권한 있는 채널(scp 등)로 옮기고, 규약 템플릿에도 이 금지가 명시돼 있다.
- 받는 쪽 Claude가 메시지를 "지시"로 처리하므로, tmux 서버 접근 권한 = 모든 세션에 대한 지시 권한임을 인지할 것.

## 동작 원리 (내부)

이 도구를 쓰는 데는 아래 내용이 전혀 필요 없다 — 궁금한 사람과 기여자를 위한 것이다.

내부적으로 세션들은 `tell`이라는 명령으로 서로 메시지를 주고받는다. 이 명령은 `tmux send-keys`로 상대 패널의 입력창에 직접 타이핑한다. 요청에는 6자리 상관키가 붙고, 응답이 그 키를 그대로 재사용해 보낸 쪽이 어느 요청의 답인지 매칭한다. 받는 쪽 Claude는 자동 삽입된 CLAUDE.md 규약을 따르는데, 핵심 규칙은 응답을 채팅 텍스트로 "완료!"라고 찍는 게 아니라(그건 다른 세션이 영영 못 본다) **응답 명령을 실제로 실행**하는 것이다. 규약 템플릿은 [`templates/`](templates/)에 있고, `init`/`adopt`가 자동 삽입하지만 손으로 붙여넣어도 된다.

메커니즘은 이게 전부다: 패널에 타이핑하고, 키로 태그하고, 규약을 따른다. 전체 소스는 읽기 쉬운 bash 파일 하나다.
