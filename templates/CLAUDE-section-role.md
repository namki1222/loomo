## tell — 세션 간 협업 (tmux 브릿지)

너는 tmux 세션 **{{SESSION}}** 의 한 역할 패널이다. 요청·응답은 **상대 채팅 입력창에 직접 메시지를 보내** 처리한다. (inbox/폴링 금지)

**현재 비서 확인:** 비서(허브)는 런타임 설정이다. `loomo hub status`를 실제 실행하면 이 명령이 **너 자신이 비서인지**를 알려준다 — 네가 비서면 `세션|역할`만 출력(exit 0)하고, 아니면 `you are … — NOT the hub`을 출력(exit≠0)한다. **출력에 'NOT the hub'이 있거나 exit가 0이 아니면 너는 비서가 아니다 → 절대 비서처럼 행동·라우팅하지 마라**(다른 세션·패널에 요청을 뿌리지 말 것, 크로스-프로젝트는 비서를 거쳐라). 표시된 비서 주소는 이 파일의 과거 주소보다 항상 우선한다.

**보내기(요청):** `tell <세션> <역할> "<자기완결 메시지>"`
- tell이 6자리 KEY를 자동 생성·출력 → 그 KEY의 응답을 기다림으로 기억.

**범위 — 요청은 네 프로젝트 안에서만:** `loomo hub status`가 너를 현재 비서(허브)로 지목하지 않는 한, **요청은 네 세션 {{SESSION}} 안의 역할에게만** 보낸다. 다른 프로젝트(세션)에 닿으려고 `tell <다른세션> ...` 하지 마라 — 대신 **비서를 거쳐라**(크로스-프로젝트 작업을 비서에게 한 번 요청하면 비서가 위임한다). **"전체 세션"·"모두"는 이 프로젝트의 패널만** 뜻하며, 등록된 모든 세션이 아니다. (`tell -r` 응답은 예외: 요청 헤더의 `from`(다른 세션의 비서 포함)에게 항상 회신한다.)

**응답하기:** `tell -r <KEY> <보낸세션> <보낸역할> "<메시지>"`
- ⚠️ 응답 = Bash로 `tell -r` 를 **실제 실행**하는 것. 채팅에 텍스트만 찍는 건 응답이 아니다 — 각 세션은 독립이라 네 텍스트 출력은 상대에게 절대 안 보인다.
- 회신 주소는 받은 요청 헤더 **`[세션 요청 - KEY from <세션>/<역할>]` 의 `from`** 에서 읽는다(자기 역할로 보내면 자기한테 가는 루프).
- `from` 이 없으면(=tmux 밖/사용자 직접) 어느 세션:역할로 답할지 사용자에게 묻는다.

**헤더 처리**
- `[세션 요청 - KEY]` 받으면: 현재 작업을 중단하지 말고, 처리 시작 시 `loomo task ack <KEY>`를 실제 실행 → 끝나면 그 KEY로 `tell -r` 응답. (생성 당시 허브: `{{HUB_SESSION}}/{{HUB_ROLE}}`, 현재값은 `loomo hub status`로 확인)
- 사용자 승인이나 확인이 필요해 멈췄다면 `loomo task status <KEY> needs_approval "<이유>"`, 실패했다면 `loomo task status <KEY> failed "<이유>"`를 실제 실행한다.
- 처리 중 **되물음·확인(설계·범위 판단)이 필요하면**: 채팅으로 사용자에게 묻지 말 것(발신자는 네 화면을 못 본다). 받은 KEY로 `tell -r <KEY> <발신세션> <발신역할> "확인 필요: <질문>"` 을 실행해 **발신자에게** 물어라 — 질문도 응답이다. 단, **하네스 권한승인**(분류기 차단 등)은 사용자만 풀 수 있으므로 사용자에게 요청하되, 그 **대기 상태를 발신자에게도 `tell -r`로 알려라**.
- `[세션 응답 - KEY]` 받으면: 내가 보낸 어느 요청의 결과인지 KEY로 매칭.
- 헤더 없는 메시지 = 사용자(사람) 직접 입력 → 미루지 말고 즉시 처리.

**보안**: 비밀번호·토큰·시크릿은 절대 tell 메시지에 평문으로 넣지 않는다(상대 패널 스크롤백에 남음).
**본문 원칙**: 다른 세션에 보내는 요청·응답 본문에는 작업 내용과 결과만 쓴다. ACK, KEY, `tell -r` 사용법 같은 loomo 내부 설명은 협업 규약에만 두고 매번 반복하지 않는다.
